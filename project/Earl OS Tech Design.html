00 Overview

What we're building, and what we're not building.

Earl OS is your daily command center for running Earl Neal Event Management Services. It pulls together everything you check in a typical work day or show day — calendar, email, weather, news, tasks, journals — into one window-manager dashboard you can rearrange to taste. It works in any modern web browser and (eventually) on your phone.

In scope for v1

  • Authenticated single-user app (just you for now, but built so you can invite crew later)
  • Real Gmail and Google Calendar integration
  • Real weather data for outdoor stops
  • News/industry feed aggregated from RSS
  • Show schedule, daily-resetting task templates, journals, notes
  • Sync across all your devices in real time
  • Works offline (with sync when you reconnect)

Out of scope (for now)

  • Multi-user collaboration (sharing show journals with crew, etc.) — easy to add in v2
  • Native iPhone/Android apps — we'll use a PWA first (more on that below)
  • AI features (auto-summarize emails, suggest tasks) — possible later, not foundational

Why three layers? The browser only knows how to draw the screen. Your data lives in the cloud so it follows you device-to-device. What software we pick for each layer, and why.

About GoDaddy: Your existing hosting is great for the portfolio site but isn't built for a real-time app like Earl OS — it doesn't support PostgreSQL, OAuth-backed APIs, or live sync. The industry-standard pattern is to keep the marketing site on its current host and run the app on a service designed for apps. That's what Vercel + Supabase do. Your earlneal.com visitors never know the difference — both sit under the same brand.
?
Why Supabase instead of Firebase or AWS? Supabase uses PostgreSQL — a real database that any developer on earth can read. If you ever want to switch services or hand the project off, you can. Firebase locks you into Google's proprietary format. AWS is overkill for one person.

04 Data model

What gets stored, and how it's organized. Each row in the table below is a "table" in the database — think of it like a spreadsheet tab.

TableWhat it holdsExample row
users Your account info id · email · name · title · status · current_event · current_venue
show_dates Every date you have a show, with details user_id · date · venue · city · is_outdoor · cap · load_in_time
task_templates The repeating checklists (Office daily, Show pre/day/post) id · user_id · kind · text · order
task_state Which template tasks are checked, for which date user_id · template_id · scope_date · done
tasks_oneoff The one-off task list (not a template) id · user_id · text · done · created_at
journal_personal One entry per calendar day user_id · date · mood · energy · grateful · win · reflection
journal_show One entry per show date user_id · show_date · venue · times · attendance · arrival · parking · load_in · meals · show · load_out · depart · general
notes The Quick Notes scratchpad (single blob) user_id · text · updated_at
news_cache Stories scraped from RSS, de-duplicated id · source · category · title · url · published_at · hot
weather_cache Recent weather lookups, so we don't hit the API every render location · forecast_json · fetched_at
i
Why two tables for tasks? Because you asked for templates that carry over. The task_templates table holds the master list — change the text once, every future show picks it up. The task_state table holds "which boxes are checked on which date" — that's how today's check-offs don't carry to tomorrow.

05 Widget-by-widget breakdown

What each window in the dashboard actually does behind the scenes.

Personal

Loads your users row. Edits write back to the same row.

Complexity: Trivial

Calendar

On load, requests today + tomorrow's events from your Google Calendar via the Calendar API. We cache the response for 5 minutes so we don't spam Google. A small badge in your calendar settings (or a "type: show" tag) lets us auto-detect which dates are show dates.

Complexity: Moderate — Google OAuth is the trickiest part of v1.

Gmail

Same OAuth as Calendar (one login, two services). We request the last ~20 unread messages, store the latest preview in our DB so the inbox loads instantly when you open Earl OS, then refresh in the background.

Complexity: Moderate

Weather

For each show in your show_dates table marked is_outdoor=true, an Edge Function pulls a forecast from Tomorrow.io once an hour and stores it in weather_cache. The widget reads from cache. Risk classification (low/med/high) is computed from temperature, precip probability, and wind speed.

Complexity: Moderate

News & Industry

A scheduled Edge Function runs every 30 minutes. It fetches each publication's RSS feed, de-duplicates against the news_cache table by URL, and inserts new stories. The widget queries the latest ~30 stories. Category is inferred from the source (or simple keyword rules).

Complexity: Easy — RSS is a 30-year-old protocol; well-supported.

Tasks (Show, Office, One-offs)

Already covered in the data model section. Templates carry over; daily state resets.

Complexity: Easy (already prototyped)

Quick Notes

Plain text in a single row. Saves debounced 1 second after you stop typing.

Complexity: Trivial

Personal Journal & Show Journal

Save on every keystroke (debounced). Each date is its own row. The Show Journal is keyed by show date, not calendar date — so the "Show #042" entry stays the same even if you reflect on it the next morning.

Complexity: Easy

Future: Export Show Journals as PDF

Edge Function generates a clean PDF from the show journal entry on demand or after every show, optionally auto-uploads to a designated Google Drive folder.

Complexity: Moderate

06 Auth & security

How you log in, and how we keep your stuff safe.

Login

One button: "Sign in with Google". Same Google account you use for Gmail/Calendar. No password to remember. This single OAuth approval covers all three: identity, Gmail, Calendar.

Where credentials live

  • You never see or store Google's password — Google handles it.
  • Supabase stores a short-lived refresh token (encrypted at rest) that lets us re-request access to Gmail/Calendar on your behalf, without you logging in again.
  • If you revoke access in your Google account settings, our access dies instantly.

Row-Level Security

Every table in Supabase has a rule that says "a row is visible only if its user_id matches the currently-signed-in user." Even if someone got a backup of the database, they couldn't read another user's data without that user's credentials.

What if your laptop is stolen?

  • Earl OS auto-locks after a configurable idle period (default: 30 min). Logging back in requires Google.
  • You can revoke a device from a "Sessions" page (we'll build this in v1).
!
Critical: Reading Gmail requires Google's "Restricted Scope" review — a one-time security review (~1-2 weeks turnaround) before the OAuth screen says "Earl OS" instead of "unverified app." We can build and test in development first; submit for review once the app is stable.

07 Sync across devices

Check a task on your laptop, see the check appear instantly on your iPad.

Supabase has a built-in feature called Realtime — when a row in the database changes, every connected device gets notified within ~100 ms. The dashboard re-renders automatically. No "pull to refresh" needed.

What this means in practice

  • Type a note on your laptop at the office → walk into the venue → open your iPad → note is there.
  • Check off "Soundcheck complete" on your phone → your laptop on the bus updates instantly.
  • If you're offline, changes queue locally and sync when you reconnect.
?
What about conflicts? If two devices edit the same field at the same instant, the last write wins. For tasks (boolean check-offs) this is essentially never a problem. For text fields (journal entries), we'll use a small "last edited 12s ago on iPhone" indicator and warn if it looks like a clash.

08 Mobile strategy

How Earl OS lives on your phone — without building a separate iPhone/Android app.

Phase 1: PWA

A Progressive Web App is a regular website that an iPhone or Android treats like an app. You visit Earl OS in Safari, tap "Add to Home Screen," and now there's an Earl OS icon on your phone. Opens in fullscreen, works offline, looks and feels like a native app.

  • Pros: Free (no App Store fees), no review process, deploy updates instantly.
  • Cons: No native push notifications on iPhone until iOS 16.4+ (we have that now, May 2026). Slightly less polish on gestures than a real native app.

Phase 2 (if needed): React Native

If we ever need push notifications, better gestures, or to be in the App Store: React Native lets us reuse 80% of our existing React code in a native-quality iPhone/Android app. We'd revisit this in 6 months once we know what's missing.

i
Plain English: Start with the website-on-your-home-screen approach. It's free, fast to ship, and 90% as good. Only build a "real app" if we discover something the PWA truly can't do.

12 Glossary

Terms a developer will throw around. Cheat-sheet so you can call BS when you need to.

API
"Application Programming Interface" — a structured way for one piece of software to talk to another. When the dashboard fetches your Gmail, it's "calling the Gmail API."
OAuth
The standard "Sign in with Google / Facebook / Apple" handshake. You give Earl OS permission to act on your behalf without ever giving us your password.
Backend
Everything that runs on a server, not in your browser. Database, login system, scheduled jobs.
Frontend
What you actually see and click on. Runs in your browser.
Database
Where data is stored long-term. Think of it as a very strict spreadsheet that many people can read and write at once.
Table / Row / Column
Database structure. A table is like a spreadsheet tab. Each row is one record (one task, one show, one email). Columns are the fields (text, done, date, etc.).
PWA
Progressive Web App. A website you can install to your phone's home screen and use offline like a real app.
Serverless / Edge Function
A small piece of code that runs only when triggered (e.g. every 30 min on a schedule, or when someone clicks a button). No always-on server to babysit.
RSS
A standard, machine-readable format that most news sites publish. Lets us pull headlines without scraping.
Realtime
When a database change pushes itself to all connected devices in milliseconds, instead of devices having to poll for updates.
Cache
Saved-for-later copy of something. Speeds things up so we don't re-fetch the same data over and over.
Row-Level Security (RLS)
A database rule that says "this user can only see rows that belong to them." Keeps your data isolated from anyone else's.
Webhook
"Call us when X happens." Used so we don't have to poll constantly — e.g. Gmail can tell us "new email arrived" instead of us asking every 30 seconds.
Geocoding
Turning a place name ("Red Rocks Amphitheater") into latitude/longitude so we can ask for its weather.
Service Worker
A small piece of code that runs in the background of your browser. Powers offline support in PWAs.

13 Next steps

If you give the green light, here's what happens in the first 48 hours.

  1. Subdomain locked in: os.earlneal.com. Echoes the product name and stays short for mobile typing.
  2. Create accounts: Supabase, Vercel, Google Cloud Console (developer side), Tomorrow.io. All free. We'll do this together via a screen share.
  3. Spin up the Supabase project and create the empty tables described in Section 4.
  4. Connect the existing prototype to Supabase — your tasks, journals, and notes start saving to the cloud instead of just your browser.
  5. Add one DNS record on GoDaddy: a CNAME for os pointing to Vercel. ~2 minutes inside GoDaddy's DNS panel; doesn't touch your existing website.
  6. Deploy to os.earlneal.com — you can now hit it from any device, and your portfolio site at www.earlneal.com is completely unaffected.

After that we're into Phase 2 (Gmail + Calendar), which is the most exciting visible upgrade.
